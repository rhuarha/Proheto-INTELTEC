1. CORREá‡áƒO DE DATA NO RECEBIMENTO / PROCESSAMENTO
==================================================

Foi identificado um bug:
- ao incluir uma ordem de produção com a data de hoje, na tela de processamento aparece a data de ontem;
- ao incluir com a data de ontem, aparece anteontem.

Corrigir o tratamento de datas para evitar deslocamento por timezone.

Regras:
- data_recebimento deve ser tratada como data local, sem conversão indevida para UTC.
- hora_recebimento deve ser tratada como hora local.
- ao salvar e ao exibir, a data deve permanecer exatamente igual á  data informada pelo usuário.
- não deve ocorrer subtração automática de um dia.
- revisar backend, frontend, ORM e serialização JSON para impedir conversão errada de datas.

Sugestão técnica:
- armazenar data_recebimento como DATE no MySQL/MariaDB.
- armazenar hora_recebimento como TIME.
- no frontend, exibir data_recebimento diretamente no formato DD/MM/YYYY ou YYYY-MM-DD sem usar conversão UTC.
- evitar new Date(data_recebimento) para datas sem hora.
- criar função utilitária para formatar datas puramente como string local.

6. SIGNIFICADO DOS CAMPOS DE clientes
==================================================

Usar as seguintes definiá§áµes no cadastro de clientes e, se possá­vel, como comentários no model/schema ou documentação interna:

nome_interno:
Nome pelo qual o cliente será reconhecido internamente no sistema.

juridica:
Define se o cliente é pessoa jurá­dica ou pessoa fá­sica.
true = pessoa jurá­dica.
false = pessoa fá­sica.

cnpj_cpf:
Se juridica = true, deve receber CNPJ.
Se juridica = false, deve receber CPF.

inscr_estadual:
Inscrição estadual do cliente.

inscr_municipal:
Inscrição municipal do cliente.

telefone:
Telefone para contato.

logradouro:
Nome da rua.

numero:
Náºmero do imóvel. Deve aceitar também a string "S/N".

complemento:
Complemento do endereá§o. Campo opcional.

bairro:
Nome do bairro do cliente.

cidade:
Nome da cidade do cliente.

cep:
Código de endereá§amento postal.

uf:
Unidade federativa, com 2 caracteres. Exemplo: RS, SC, SP.

email_nfse:
E-mail para envio de NFSe.

nome_contato:
Nome da pessoa de contato.

email_contato:
E-mail da pessoa de contato.

tipo_faturamento:
N = Nota Fiscal / NFSe.
R = Recibo.

emite_boleto:
Define se o cliente exige boleto para pagamento.

exige_demonstrativo:
Define se o cliente exige demonstrativo para faturamento.

email_aprova_demonstrativo:
E-mail usado para aprovação do demonstrativo.

email_informa_produto_embalado:
E-mail usado para informar que o produto está embalado e disponá­vel para retirada.

pedido_compra:
Define se é necessário pedido de compra emitido pelo cliente para emissão da NFSe.

diretorio_producao:
Diretório onde serão salvos os recebimentos de ordem de produção.

diretorio_demonstrativo:
Diretório onde serão salvos os demonstrativos.

prazo_pagamento:
Prazo para pagamento em dias.

tipo_fechamento:
Define o tipo de fechamento do cliente:
- IMEDIATO
- POR_VALOR
- POR_VALOR_OU_PRAZO
- POR_PRAZO
- MENSAL_FIXO

valor_alvo:
Valor que, quando atingido, dispara gatilho futuro para emissão de demonstrativo.

valor_minimo:
Valor má­nimo para faturamento.

prazo_maximo_dias:
Prazo máximo em dias para acumular ordens de produção sem faturar.

dia_fechamento:
Dia ideal para fechamento.
Exemplos:
- primeiro dia áºtil do mês
- dia 15
- áºltimo dia áºtil do mês

fechar_ao_disponibilizar:
Define se o fechamento deve ocorrer tão logo o trabalho seja disponibilizado para retirada.
